- 日期：`2026-04-30`

## 一、先定原则

这次重构必须严格遵守以下原则：

1. **不做资源系统**
   - 不新增“资源中心”
   - 不做附件上传/管理平台
   - 不做复杂资料树
   - 所谓“附件”“提示词文档”“skills”都只作为 AI 调用时的项目内资源来源

2. **流程优先，模板降级**
   - 当前产品里 Prompt 模板权重过高
   - 后续产品主入口不应该是模板，而应该是“小说流程”
   - 模板只作为阶段动作的底层配置

3. **提示词分两类**
   - `可见提示词`：允许用户在界面里查看和编辑
   - `隐藏提示词`：仅内部调用，不直接暴露给用户

4. **提示词必须项目化**
   - 不同小说应有不同提示词组合
   - 不能继续用一套全局固定模板去覆盖所有小说
   - 提示词配置应挂在项目上，而不是挂在全局设置上

5. **skills 只允许项目级**
   - 可以下载 skill
   - 但必须只放在当前项目目录
   - 不做全局 skill 安装

## 二、当前项目存在的主要问题

## 2.1 产品问题

当前项目的问题不是“功能不够多”，而是：

- 主入口不对
- 工作流不清楚
- 小说写作过程没有被编排
- Prompt 模板功能过早做重

作者现在进入项目后，会看到：

- 概览
- 世界观
- 角色
- 关系
- 大纲
- 章节
- 设置

这些模块本身都能工作，但问题是：

- 作者不知道现在该先做哪一步
- 作者不知道这一阶段该调用什么动作
- 作者不知道哪些提示词该自己看，哪些不该看

## 2.2 Prompt 功能问题

当前已经做了项目级 Prompt 模板覆盖与复制，但它已经开始偏重：

- 模板过多
- 模板暴露层级过深
- 用户心智被带到“调 prompt”，而不是“推进小说流程”

这和 `1.md / 2.md / 3.md` 的核心思想不一致。

真正需要的是：

- 在正确阶段触发少量高频动作
- 提示词根据项目动态变化
- 用户只看到需要看的那部分

## 2.3 附件认知问题

当前容易误把附件理解成：

- 用户要管理的资源

但根据你的要求，正确理解应该是：

- **附件 = AI 内部调用时的提示词素材 / 流程素材 / 技能素材**

所以它们不应该先进入一个“资源管理系统”，而应该进入：

- 项目流程配置
- 项目提示词上下文
- 项目 skill 挂载

## 三、重构后的目标产品形态

## 3.1 新的产品主入口

新增一个顶级工作台面板：

- `小说流程`

它将成为小说项目的主入口。

现有模块不删除，但降为“执行层”：

- 世界观
- 角色图鉴
- 关系与组织
- 剧情大纲
- 章节创作
- 设置

小说流程面板作为“调度层”。

## 3.2 小说流程面板职责

小说流程面板只负责：

1. 显示当前阶段
2. 显示当前阶段目标
3. 显示当前阶段可用动作
4. 跳转到对应执行模块

它**不负责**：

- 编辑所有数据
- 展示全部 Prompt 细节
- 做复杂资源管理

## 四、MVP 功能结构

## 4.1 固定 5 个阶段

根据 `1.md / 2.md / 3.md`，MVP 固定为：

1. 选题与参考
2. 故事立项
3. 设定搭建
4. 大纲与章节规划
5. 正文创作与优化

每个阶段只展示：

- 阶段说明
- 当前状态
- 建议动作
- 跳转入口

## 4.2 每个阶段的动作类型

### 阶段 1：选题与参考

动作：

- 提炼平台风格
- 提炼参考书风格
- 提炼题材爆点

说明：

- 这类动作的提示词可以部分展示给用户
- 但内部“资料拼接规则”不展示

### 阶段 2：故事立项

动作：

- 生成故事背景
- 生成主角路线
- 生成外挂 / 金手指设定

### 阶段 3：设定搭建

动作：

- 生成势力列表
- 生成角色骨架
- 生成人物关系盘
- 生成关键设定清单

### 阶段 4：大纲与章节规划

动作：

- 批量补本卷节点
- 生成后续剧情链
- 调整分卷节奏

### 阶段 5：正文创作与优化

动作：

- 本章初稿
- 续写片段
- 章节分析
- 去 AI 味优化

## 五、提示词系统重构方案

## 5.1 明确分层

把提示词分成 3 层：

### A. 系统内核提示词

作用：

- 控制 AI 基本行为
- 拼接附件内容
- 拼接流程约束

特点：

- 用户不可见
- 不在设置页直接编辑

### B. 项目流程提示词

作用：

- 与小说项目绑定
- 由不同小说的题材、站点、风格决定

特点：

- 项目级
- 可部分展示
- 可配置

### C. 用户可见动作提示词

作用：

- 对应阶段动作卡
- 用户能看见动作名和主要 prompt

特点：

- 可编辑
- 可复制
- 不暴露底层拼接逻辑

---

## 5.2 哪些提示词可以给用户看

建议可见：

- 风格提炼 prompt
- 故事背景生成 prompt
- 人物骨架 prompt
- 本章初稿 prompt
- 章节分析 prompt
- 去 AI 味优化 prompt

建议不可见：

- 系统身份设定大段规则
- 文件拼接逻辑
- skills 路径注入逻辑
- 联网考据规则拼接逻辑
- 项目级内部约束融合逻辑

原因：

- 用户真正需要调整的是“动作语义”
- 不需要承担底层框架复杂度

---

## 5.3 当前 Prompt 模板功能要怎么改

不是删除，而是降级重构。

### 保留

- 项目级模板覆盖
- 复制为新模板
- 删除项目模板

### 修改

- 不再把它叫做“Prompt 模板主系统”
- 改为“阶段动作配置”

### 隐藏

- 当前设置页里不应再完整铺出所有模板为主入口
- 后续应只展示：
  - 当前项目常用动作
  - 当前阶段动作

### 结论

当前已做的模板能力可以保留底层实现，但 UI 入口和产品定位必须调整。

## 六、skills 方案

## 6.1 目录规则

在项目根目录下新增：

- `.project-skills/`

只识别这个目录中的项目级 skills。

## 6.2 支持内容

MVP 只支持：

- 识别项目内 skills
- 展示 skill 列表
- 标记启用状态
- 将 skill 作为 AI 内部流程资源调用

## 6.3 不支持内容

- 全局 skill 安装
- 跨项目共享 skill
- skill 市场
- 自动联网安装 skill

## 6.4 “下载 skill”怎么处理

你要求“阅读文章可以自主下载 skills，下载 skills 只放在这个项目中”。

所以功能方案应为：

- 产品只提供 `项目级 skills 目录识别`
- 下载动作不先做成平台功能
- 用户把下载好的 skill 放进 `.project-skills/`
- 产品自动识别

这样最小，也最稳。

## 十、建议开发顺序

### Phase 1

1. 新增 `小说流程` 面板
2. 固定 5 个阶段卡
3. 阶段卡支持跳转现有模块

### Phase 2

1. 增加项目级阶段状态持久化
2. 增加阶段动作卡
3. 将当前高频 AI 动作挂到阶段中

### Phase 3

1. 增加项目级 skills 目录识别
2. 展示 skills 列表
3. 将 skills 作为 AI 内部调用资源

### Phase 4

1. 将当前 Prompt 模板设置页降级为“动作配置”
2. 只保留高频可见模板
3. 隐藏系统级提示词逻辑

## 十一、给 AI 执行时的明确指令

后续任何 AI 按本方案开发时，必须遵守：

1. 不新增资源系统

6. 项目级提示词分为：
   - 可见动作提示词
   - 内部系统提示词
7. 下载到的 skills 只放项目目录 `.project-skills/`

## 十二、最后结论

接下来改这个项目，正确顺序不是：

- 继续加模板
- 继续加零散按钮
- 继续做大资源系统

而是：

1. 先做 `小说流程` 面板
2. 用它把现有模块串起来
3. 把附件、提示词、skills 都降为项目级流程资源
4. 再按阶段慢慢补动作

这才是符合 `1.md / 2.md / 3.md` 主体逻辑，也最适合当前项目的最小实现方案。
