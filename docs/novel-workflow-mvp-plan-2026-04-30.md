## 一、纠偏后的核心判断

上一版方案把重点放偏了。

真正最重要的不是附件里的项目样例文件，而是：

1. `1.md` 讲的是：
   - 用文件当持久记忆
   - 用规划文件维持上下文
   - 用渐进式加载控制正文、资料、考据和审查
   - 本质上是 **小说项目工作方法论**

2. `2.md` 讲的是：
   - 如何从平台选题开始
   - 如何下载参考小说
   - 如何做风格提炼、故事背景、体系、角色、势力、正文分卷
   - 如何把 skill / 指令嵌进自己的项目
   - 本质上是 **小说生产流水线**

3. `3.md` 讲的是：
   - 写作规划
   - 风格内化
   - 文本优化
   - 去 AI 味
   - 人物构建、代入感、节奏、开篇、爽点、包装
   - 本质上是 **写作能力层**

所以 CharacterArc 的最小 MVP 不应该先做：

- 大 Prompt 模板工坊
- 复杂流程编排器
- 全局 skills 市场

而应该先做：

- **项目级小说工作台**
- **项目级参考资料管理**
- **项目级 skills 挂载**
- **少量高频动作卡**

---

## 二、从 1.md / 2.md / 3.md 提炼出的产品目标

CharacterArc 应该先解决的核心问题不是“我有多少个 AI 按钮”，而是：

> 我现在在写一本小说，当前处于哪个阶段，需要读哪些资料，用哪类动作，产物应该落到哪里。

这件事可以拆成 4 个最小目标：

1. 作者能把一本书的参考资料、设定、正文、计划文件都挂在同一个项目里。
2. 作者能按清晰阶段推进，不会在一堆模块之间迷路。
3. 作者能在每个阶段调用少量高频动作，而不是面对一大坨模板。
4. 作者能把项目自己的 skills 放在项目内部，而不是污染全局环境。

---

## 三、MVP 的正确定位

MVP 不是“万能写作平台”，而是：

- 一个以 **项目文件** 为核心的小说工作台
- 一个以 **阶段推进** 为骨架的创作流程层
- 一个以 **项目级 skills** 为边界的扩展机制

一句话：

> 先把“写一本小说的流程”跑顺，再谈更复杂的提示词系统。

---

## 四、MVP 应该做什么

## 4.1 小说项目工作台

新增一个真正围绕小说生产的总入口，建议名字就叫：

- `小说流程`
  或
- `写作流程`

这个入口不替代现有模块，而是把它们串起来。

这个面板里只做 3 件事：

1. 显示当前阶段
2. 显示当前阶段所需资料
3. 显示当前阶段的高频动作

---

## 4.2 固定 5 个阶段，别做太多

根据 `2.md` 和 `3.md`，最小版本建议固定为这 5 个阶段：

1. 选题与参考
2. 故事立项
3. 设定搭建
4. 大纲与章节规划
5. 正文创作与优化

为什么不是更多阶段：

- 因为这是 MVP
- 太细会重新变复杂
- 这些阶段已经够覆盖 1/2/3.md 里的主流程

每个阶段只需要 4 个字段：

- 当前目标
- 依赖资料
- 建议动作
- 产物落位

---

## 4.3 项目资料挂载

这是 MVP 里最重要的能力之一。

从 `1.md` 看，文件不是附件，而是工作记忆。
从 `2.md` 看，参考小说和项目资料本来就要进项目。

所以 MVP 里必须支持：

- 项目内参考资料目录
- 项目内设定文件目录
- 项目内正文卷目录
- 项目内计划文件目录

最小能力：

1. 能在项目中登记资料文件
2. 能按类别查看资料文件
3. 能一键打开
4. 能在某个阶段里标记“本阶段必读资料”

第一批建议支持的文件类型：

- `.md`
- `.txt`
- `.docx`

第一批建议资料分类：

- 参考小说
- 项目设定
- 人物关系
- 任务计划
- 正文卷
- 提示词 / 指令

---

## 4.4 项目级 skills

这是你明确要求的点，必须单独拎出来。

规则必须非常清楚：

- skills 可以下载
- 但只能放在当前项目中
- 不允许默认装到全局

所以 MVP 方案是：

### 目录规则

在项目根目录下新增一个专用目录，例如：

- `.project-skills/`

或

- `novel-skills/`

推荐用：

- `.project-skills/`

因为更明确是项目私有能力。

### MVP 能力

只做这几件事：

1. 识别项目目录下的 skills
2. 展示 skill 名称、路径、说明
3. 支持手动启用 / 停用
4. 支持把外部下载到的 skill 拷进项目目录

### MVP 不做

- 全局技能安装
- 技能市场
- 自动从远程商店安装
- 多项目共享技能

---

## 4.5 动作卡，而不是 Prompt 工坊

`2.md` 和 `3.md` 的重点不是让作者自己维护几十个 prompt，而是：

- 在不同阶段做对的事情

所以 MVP 里不要再继续扩张“模板工坊”，而要先收成：

- **阶段动作卡**

每个阶段只保留少量高频动作。

### 阶段一：选题与参考

动作建议：

- 提炼参考书风格
- 提炼题材爆点
- 提炼平台读者偏好

### 阶段二：故事立项

动作建议：

- 生成故事背景
- 生成主角路线
- 生成金手指 / 外挂设定

### 阶段三：设定搭建

动作建议：

- 生成势力列表
- 生成角色骨架
- 生成人物关系盘
- 生成关键设定清单

### 阶段四：大纲与章节规划

动作建议：

- 批量补本卷节点
- 生成后续剧情链
- 调整分卷节奏

### 阶段五：正文创作与优化

动作建议：

- 本章初稿
- 续写片段
- 章节分析
- 去 AI 味优化

这些动作本质上还是 prompt，但在产品层面不再叫“模板工坊”，而是叫：

- `阶段动作`

这样更符合真实使用心智。

## 五、MVP 的最小数据结构

## 5.1 小说流程阶段

建议新增一个非常轻的结构：

```ts
type NovelWorkflowStage = {
  id: 'reference' | 'premise' | 'setting' | 'outline' | 'draft'
  title: string
  description: string
  status: 'todo' | 'doing' | 'done'
  relatedFilePaths: string[]
  quickActionIds: string[]
  targetPanel?: PanelName
}
```

## 5.2 项目级资料项

```ts
type ProjectResourceItem = {
  id: string
  path: string
  label: string
  type: 'reference' | 'setting' | 'plan' | 'chapter' | 'prompt' | 'skill'
}
```

## 5.3 项目级 skill

```ts
type ProjectSkillItem = {
  id: string
  name: string
  path: string
  enabled: boolean
  description?: string
}
```

这里的重点是：

- 先轻
- 先够用
- 不做复杂 DSL

## 六、MVP 不该做什么

基于 1/2/3.md，以下内容现在都不该优先做：

- 大型 Prompt 模板平台
- Prompt 分类树
- 模板市场
- 全局 skills 安装器
- 自动联网拉取技能仓库
- 复杂附件知识图谱
- 跨项目知识共享
- 用户体系 / 云同步

原因很简单：

- 都会把小说流程 MVP 做重
- 会让产品继续冗余
- 不符合你现在要的“先做最小可用”

---

## 七、推荐开发顺序

### 第 1 步：小说流程壳

先做：

- 新增 `小说流程` 面板
- 固定 5 个阶段卡
- 阶段卡支持状态切换
- 阶段卡能跳转到现有模块

### 第 2 步：资料挂载

再做：

- 项目资料列表
- 阶段绑定资料
- 打开资料

### 第 3 步：阶段动作卡

再做：

- 每阶段挂少量动作
- 动作调用现有 AI 能力

### 第 4 步：项目级 skills

最后做：

- 识别 `.project-skills/`
- 展示项目 skills
- 启用 / 停用
- 支持把下载好的 skill 拷入项目目录

---

## 八、针对当前产品的直接建议

### 1. Prompt 模板功能降为底层能力

现在这个功能不该成为主入口。

它应该退到：

- 阶段动作卡的底层配置

### 2. 小说流程面板成为作者主入口

如果用户是来写小说，第一眼最该看到的是：

- 现在处于哪个阶段
- 下一步该做什么
- 需要打开哪些资料

而不是：

- 先去设置页改模板

### 3. 项目级 skills 明确边界

只认项目目录里的 skills，不搞全局。

---

## 九、建议立刻开发的 MVP 清单

就做这些，够了：

1. `小说流程` 面板
2. 固定 5 个阶段卡
3. 每阶段支持：
   - 状态切换
   - 跳转现有模块
   - 绑定资料文件
4. 项目资料列表
5. 项目级 skills 目录识别
6. 阶段动作卡先挂现有高频动作

---

## 十、最终结论

以 `1.md / 2.md / 3.md` 为主依据后，CharacterArc 的正确方向很清楚：

- 不是先做更复杂的 Prompt 模板系统
- 不是先做更大的技能市场
- 而是先做一个 **项目级小说流程工作台**

MVP 的关键不是“功能多”，而是：

- 阶段清楚
- 资料就位
- 动作够用
- skills 项目私有

这才是最贴近你当前材料、也最适合继续开发的方案。
