# 审查结论与问题记录

## 第1章开篇审查结论

### 1. 问题：第1章原始设计吃得太多

- 证据：同时想完成大破局、重反杀、系统大说明、资源大到账、章末大跳级。
- 影响：开书会显得急、飘、假，后续两章反而没梯子。
- 最小修法：第1章只做临时脱锅、小钱到账、关键页到手、系统初显和钩到大头上面那条线。

### 2. 问题：大头如果把真钱、真账和全部把柄都摆出来，会显得过蠢

- 证据：这种写法会让反派像是专门给主角送装备。
- 影响：冲突不成立，凌夜的破局也会失去含金量。
- 最小修法：大头桌上只放誊抄账页、散钱或假口径用的材料，真正原始交数页只能局部漏出一张。

### 3. 问题：凌夜不能靠“刚穿越就全知全能”破局

- 证据：如果没有抓手，读者只会觉得角色在替作者念答案。
- 影响：人物不真，压迫感会垮。
- 最小修法：必须用账页漏洞、封条号缺失、散钱形式、大头看表、堵门站位、马仔反应这些硬抓手支撑判断。

### 4. 问题：控场不能只靠一记狠手

- 证据：单纯打狠，不足以让一屋子人都停手。
- 影响：场面会变成“别人为了让主角赢而不动”。
- 最小修法：动手之后，立刻把责任链抛回去，让所有人都意识到谁先动谁先担锅。

### 5. 问题：系统第一次出现如果太重，会抢掉开篇的人物戏

- 证据：开书真正要立的是凌夜，不是系统说明书。
- 影响：第1章容易变味成系统文模板开场。
- 最小修法：系统只短促提示一次，只给实用奖励，不长篇解释。

### 6. 问题：章末钩子不能跳太高

- 证据：当前位阶还只是大头这一层。
- 影响：如果一下子跳到高层大佬，会让后续过渡断层。
- 最小修法：只钩到“大头上面那条线”，让第2章顺势接大头余波。

## 当前项目级结论

- 当前不存在正式正文，因此任何“已发生剧情”都必须谨慎入账。
- `57小说/` 只能借写法机制，不得借人物、桥段和事实。
- `.codex` 默认负责审查、规划和维护，不默认代写整章正文。
- `.gemini` 默认负责最小读档后的正文成品输出。
